Modul 12

Responsive Design

Layouts, die auf jedem Bildschirm funktionieren — von der Smartwatch bis zum Ultrawide-Monitor. Nicht nur „kleiner machen", sondern bewusst für verschiedene Kontexte gestalten.

45 Minuten
Praxisaufgabe: ~15 Min.
Lernziele
  • Mobile-First als Designstrategie verstehen und anwenden
  • Sinnvolle Breakpoints wählen — basierend auf Inhalten, nicht auf Geräten
  • Fluid vs. Fixed Layouts unterscheiden und gezielt einsetzen
  • Touch-Targets und Typografie für mobile Kontexte optimieren
  • Content-Priorisierung: Was zeigst du wann und wo?
Kerngedanke

Responsive Design bedeutet nicht, das gleiche Layout einfach zu schrumpfen. Es bedeutet, jede Bildschirmgröße als eigenständigen Kontext zu behandeln — mit angepasster Hierarchie, Proportionen und Interaktion.

Grundlagen

Responsive Design — Mehr als nur Media Queries

Viele Entwickler setzen Responsive Design mit Media Queries gleich. Aber Breakpoints allein machen noch kein gutes responsives Layout. Responsive Design ist eine Denkweise: Du gestaltest Inhalte so, dass sie in jedem Kontext verständlich, zugänglich und visuell stimmig sind. Das bedeutet, dass sich nicht nur die Breite ändert — sondern auch Proportionen, Abstände, Schriftgrößen und die Informationshierarchie selbst.

Die 5 Säulen von Responsive Design

Mobile First

Beginne mit dem kleinsten Viewport und erweitere dann. Mobile zwingt dich, Inhalte zu priorisieren — und das ist gut. Desktop bekommt dann mehr Platz für zusätzliche Informationen, nicht für Dekoration.

Content-basierte Breakpoints

Setze Breakpoints nicht bei „iPhone" oder „iPad", sondern dort, wo dein Layout bricht. Wenn Karten zu schmal werden oder Text zu lang wird — dort brauchst du einen Breakpoint. Geräte ändern sich, dein Inhalt bleibt.

Fluid + Fixed kombinieren

Nicht alles muss sich proportional zur Bildschirmbreite verhalten. Sidebars, Icons und Touch-Targets haben oft feste Größen — der Hauptinhalt füllt den Rest flexibel.

Proportionen sind nicht skalierbar

Eine Headline, die auf Desktop 48px groß ist, darf auf Mobile nicht einfach 48px × 0.5 = 24px werden. Die Verhältnisse zwischen Elementen ändern sich — große Elemente schrumpfen schneller als kleine.

Touch vs. Cursor

Touch-Targets müssen mindestens 44×44px groß sein (Apple HIG) oder 48×48dp (Material Design). Ein 32px-Button, der mit der Maus gut funktioniert, frustriert auf dem Smartphone.

Navigation: Desktop-Links vs. Mobile-Menü

Eine der häufigsten Responsive-Aufgaben: die Navigation. Auf Desktop gibt es Platz für horizontale Links. Auf Mobile muss die Navigation in ein kompaktes Format — aber ein Hamburger-Menü ist nicht immer die einzige Lösung. Entscheidend ist, dass die wichtigsten Aktionen sichtbar bleiben.

Beispiel 03

Typografie: Proportional schrumpfen vs. bewusst anpassen

Ein verbreiteter Fehler: Alle Schriftgrößen einfach proportional verkleinern. Das Ergebnis sind Headlines, die auf Mobile immer noch zu groß sind, und Body-Text, der zu klein wird. Große Elemente müssen schneller schrumpfen als kleine — das Verhältnis ändert sich.

Analyse

Im „Vorher"-Beispiel wird alles mit demselben Faktor skaliert (× 0.5). Die Headline ist immer noch dominant, aber der Body-Text wird unlesbar klein. Im „Nachher"-Beispiel wird jede Schriftstufe einzeln angepasst: Die H1 schrumpft deutlich (48 → 28px), aber der Body-Text bleibt fast gleich (16 → 15px).

Tipp: font-size: clamp(28px, 4vw, 48px) lässt die Schriftgröße flüssig zwischen Mobile und Desktop skalieren.

Regel

Skaliere große Elemente stärker als kleine. Der Unterschied zwischen Hierarchy-Stufen wird auf kleinen Bildschirmen geringer — nicht einfach proportional verkleinert.

Touch Targets: Desktop-Buttons vs. Mobile-Buttons

Auf Desktop klickst du mit einem präzisen Cursor. Auf Mobile tippst du mit einem Finger, der ca. 44–48px Fläche abdeckt. Buttons und interaktive Elemente, die auf Desktop perfekt funktionieren, können auf Mobile frustrierend klein sein — besonders wenn sie zu nah beieinander liegen.

Im „Vorher"-Beispiel liegen die Buttons nebeneinander mit nur 4px Abstand — der Nutzer tippt leicht auf das falsche Element. Beispiel 05

Sidebar-Layout: Feste Sidebar vs. Adaptives Layout

Sidebars sind auf Desktop ein bewährtes Muster für Navigation oder Filter. Auf kleineren Screens nehmen sie aber wertvollen Platz ein. Die Lösung: Gib der Sidebar eine feste Breite, dem Content den Rest — und auf Mobile wird die Sidebar zu einem Overlay oder Tab.

Analyse

Im „Vorher"-Beispiel sind sowohl Sidebar als auch Content %-basiert. Wenn der Viewport schrumpft, wird die Sidebar so schmal, dass ihre Inhalte abgeschnitten oder unleserlich werden. Im „Nachher"-Beispiel hat die Sidebar eine feste Breite (z.B. 240px), und der Content füllt den Rest mit flex: 1.

Auf Mobile verschwindet die Sidebar und wird zu einem Drawer oder Bottom Sheet, das bei Bedarf eingeblendet wird.

Regel

Nicht alles muss fluid sein. Gib Sidebars, Icons und Controls eine feste Breite — und lass den Hauptinhalt den verfügbaren Rest füllen.

Praxisaufgabe

Dashboard-Karte responsiv gestalten

Der folgende Code zeigt eine Dashboard-Karte, die nur für Desktop gestaltet ist. Deine Aufgabe: Mache sie mit CSS Media Queries responsiv — ohne die HTML-Struktur zu verändern.

Aufgabenstellung

Du bekommst eine Dashboard-Karte mit Header, Statistik-Grid und Aktions-Buttons. Aktuell funktioniert sie nur auf breiten Screens. Verbessere das CSS, sodass das Layout auf allen Bildschirmgrößen gut funktioniert.

  • Erstelle mindestens zwei Breakpoints: einen für Tablets (~768px) und einen für Smartphones (~480px)
  • Auf Tablets: Ändere das 4-spaltige Statistik-Grid zu 2 Spalten
  • Auf Smartphones: Ändere das Grid zu 1 Spalte und mache Buttons full-width
  • Passe die Headline-Größe an: Desktop 28px → Tablet 22px → Mobile 18px
  • Stelle sicher, dass alle Buttons mindestens 44px hoch sind auf Mobile
  • Bonus: Nutze clamp() für eine fluid Schriftgröße statt fester Breakpoints
